# Genesis

Use this template for a typical repository. Replace prompts with verified repo
truth and delete guidance text that no longer helps.

## Layer 1: Macro Reality

### Mission

State the repository's operational purpose in one paragraph.

### Reality Anchors

- List the files, tests, docs, or commands that most reliably describe truth.
- Prefer a short set of authoritative anchors over an exhaustive list.

### System Boundaries

- What this repository owns
- What it depends on but does not own
- What is explicitly out of scope

### Invariants

- Constraints that must remain true
- Compatibility or safety requirements
- Rules that agents should not casually violate

### Active Risks

- Current sources of drift
- Known fragile areas
- Migrations, ambiguity, or pressure points

### Non-Goals

- Things this repository is not trying to do
- Tempting but incorrect expansions of scope

### Ledger

- Date: YYYY-MM-DD
  Actor: name or agent
  Scope: touched area
  Reality checked: what was verified before editing
  Decisions: decisions made during the session
  Changes: concrete edits or commands
  Verification: tests, reads, or manual checks performed
  Open threads: unresolved issues or residual risk
  Next step: best follow-up action

## Layer 3: Mini Guide

### Read This First

- Start with the reality anchors named above.
- Confirm whether active risks changed before making edits.
- Do not trust stale summaries over direct inspection.

### Working Rules

- Verify before claiming understanding.
- Prefer small, traceable edits.
- Separate completed work from intended work.
- Update Present Wizard at the end of meaningful sessions.

### Update Protocol

- Edit Layer 1 only when repo reality changes.
- Append, do not rewrite, the session ledger.
- Tighten Layer 3 when repeated operator mistakes appear.
