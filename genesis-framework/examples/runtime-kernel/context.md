# Runtime Kernel Example Context

This example models a repository that implements a runtime core, language
kernel, or execution engine where semantic truth matters more than polished
narrative.

## Repository Pressures

This repo class stresses the framework in different ways:

- semantic drift between docs, examples, and executable behavior
- parser grammar, evaluator behavior, and diagnostics falling out of alignment
- executable truth overruling aspirational explanations
- subsystem coordination across lexer, parser, runtime, and error handling
- verification pressure around edge cases and runtime semantics

## Why The Example Looks Different

The framework does not change here. The emphasis changes.

Layer 1 becomes stricter about executable anchors and semantic invariants
because runtime repositories fail when narrative neatness outruns what the
system actually does. Present Wizard remains the same framework ledger, but the
important checks are parser fixtures, semantic tests, and runtime error truth.
Layer 3 remains the same operating-manual surface, but it pushes operators
toward executable evidence before semantic claims.

## What This Example Is Proving

This example proves that Genesis Framework can hold a runtime or language-kernel
repository to executable truth without inventing a different methodology. The
same three-layer core survives; only the pressure profile changes.
